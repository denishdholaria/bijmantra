## BrAPI Multi-Tenancy Analysis

### Key Question

**Does BrAPI have built-in multi-tenancy support?**

### Answer: NO - BrAPI is designed for single organization by default

#### Evidence from BrAPI v2.1 Specification

**1. ServerInfo has ONE organization:**

```yaml
# From BrAPI-Core/ServerInfo/Schemas/ServerInfo.yaml
organizationName: "The Institute"
organizationURL: "institute.org/home"
contactEmail: "contact@institute.org"
```

- Each BrAPI server represents **one organization**
- The `/serverinfo` endpoint returns a single organization name/URL
- No concept of multiple organizations per server

**2. No built-in multi-tenancy:**

- BrAPI spec has **NO** `organizationId` field in any entity
- Programs, Trials, Studies, Germplasm - none have organization isolation
- BrAPI assumes the entire server belongs to one organization

**3. Program is the top-level entity:**

- `Program` is the highest organizational unit in BrAPI
- Multiple programs can exist, but all belong to the same organization
- Example: ICAR-IARI might have multiple breeding programs (wheat, rice, maize)

### BrAPI's Intended Architecture

```
┌─────────────────────────────────────────┐
│  BrAPI Server (Single Organization)    │
│  Organization: ICAR-IARI                │
│                                          │
│  ┌────────────────────────────────┐    │
│  │  Program: Wheat Breeding       │    │
│  │  ├─ Trial 1                    │    │
│  │  └─ Trial 2                    │    │
│  └────────────────────────────────┘    │
│                                          │
│  ┌────────────────────────────────┐    │
│  │  Program: Rice Breeding        │    │
│  │  ├─ Trial 1                    │    │
│  │  └─ Trial 2                    │    │
│  └────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

**BrAPI's Design Philosophy:**

- One BrAPI server = One organization
- If you have multiple organizations, you deploy **multiple BrAPI servers**
- Organizations can federate by connecting to each other's BrAPI servers

## Multi-Tenancy Models Comparison

### Model 1: Single Database, Shared Schema

```
┌─────────────────────────────────────────┐
│      Single PostgreSQL Database         │
│                                          │
│  ┌────────────────────────────────┐    │
│  │  All Tables Have:              │    │
│  │  - organization_id (FK)        │    │
│  │  - tenant_id (for isolation)   │    │
│  └────────────────────────────────┘    │
│                                          │
│  Organizations:                          │
│  ├─ Org A (ICAR-IARI)                   │
│  ├─ Org B (University XYZ)              │
│  └─ Org C (Private Breeder)             │
└─────────────────────────────────────────┘
```

**Pros:**

- ✅ Simple to implement and maintain
- ✅ Cost-effective (single database)
- ✅ Easy to share reference data (traits, methods)
- ✅ Good for small to medium breeding programs
- ✅ Row-level security (RLS) in PostgreSQL

**Cons:**

- ⚠️ All tenants share same database (security concern)
- ⚠️ One tenant's heavy load affects others
- ⚠️ Backup/restore is all-or-nothing

**Best for:** Small to medium breeding organizations, research institutions

### Model 2: Single Database, Separate Schemas

```
┌─────────────────────────────────────────┐
│      Single PostgreSQL Database         │
│                                          │
│  ┌──────────────┐  ┌──────────────┐    │
│  │ Schema: OrgA │  │ Schema: OrgB │    │
│  │              │  │              │    │
│  │ - programs   │  │ - programs   │    │
│  │ - trials     │  │ - trials     │    │
│  │ - studies    │  │ - studies    │    │
│  └──────────────┘  └──────────────┘    │
└─────────────────────────────────────────┘
```

**Pros:**

- ✅ Better data isolation
- ✅ Can customize schema per tenant
- ✅ Easier to backup individual tenants

**Cons:**

- ⚠️ More complex migrations (run for each schema)
- ⚠️ Harder to share reference data
- ⚠️ Still shares database resources

**Best for:** Medium-sized organizations with different needs

### Model 3: Database Per Tenant (Federated - RECOMMENDED)

```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  DB: OrgA    │  │  DB: OrgB    │  │  DB: OrgC    │
│              │  │              │  │              │
│ - programs   │  │ - programs   │  │ - programs   │
│ - trials     │  │ - trials     │  │ - trials     │
│ - studies    │  │ - studies    │  │ - studies    │
└──────────────┘  └──────────────┘  └──────────────┘
```

**Pros:**

- ✅ Complete data isolation
- ✅ Independent scaling
- ✅ Easy to backup/restore per tenant
- ✅ Can move tenants to different servers
- ✅ True to BrAPI's federated design philosophy
- ✅ Data sovereignty (organizations own their data)

**Cons:**

- ⚠️ Complex to manage many databases (but each org manages their own)
- ⚠️ Higher infrastructure costs (but distributed)
- ⚠️ Migrations must run on all databases (but independently)

**Best for:** Large enterprise, federated network, SaaS with many independent organizations

## Federated Network Architecture

### **CHOSEN APPROACH: Federated Network of Organizations**

This is **architecturally the most elegant** and **true to BrAPI's design philosophy**.

### Architecture Overview

```
┌─────────────────────────────────────────────────────────────┐
│                    Bijmantra Ecosystem                      │
│                                                              │
│  ┌──────────────────┐  ┌──────────────────┐  ┌───────────┐│
│  │ Bijmantra Server │  │ Bijmantra Server │  │ Bijmantra ││
│  │   ICAR-IARI      │  │  Univ. XYZ       │  │  Pvt. Co. ││
│  │                  │  │                  │  │           ││
│  │ BrAPI v2.1       │  │ BrAPI v2.1       │  │ BrAPI v2.1││
│  │ Endpoints        │  │ Endpoints        │  │ Endpoints ││
│  └────────┬─────────┘  └────────┬─────────┘  └─────┬─────┘│
│           │                     │                    │      │
│           └─────────────────────┼────────────────────┘      │
│                                 │                           │
│                    ┌────────────▼──────────────┐           │
│                    │  BrAPI Federation Layer   │           │
│                    │  (Optional Discovery)     │           │
│                    └───────────────────────────┘           │
└─────────────────────────────────────────────────────────────┘
```

### How It Works

**1. Each Organization Gets Their Own Instance**

- ICAR-IARI deploys Bijmantra on their infrastructure
- University XYZ deploys on theirs
- Private companies deploy on theirs or cloud

**2. Data Sovereignty**

- Each organization owns and controls their data
- No shared database = no data leakage concerns
- Complies with data protection regulations

**3. Federation (Optional Collaboration)**

- Organizations can **choose** to share data via BrAPI
- Example: ICAR shares trait definitions with universities
- Example: Two organizations collaborate on a multi-location trial

**4. BrAPI Compliance**

- Each server is 100% BrAPI v2.1 compliant
- Can interoperate with ANY BrAPI client/server
- Future-proof for BrAPI updates

### User Management Within Each Organization

```
Organization (e.g., ICAR-IARI)
├── Admin (full access, user management)
├── Breeder (create programs, trials, crosses)
├── Technician (field data collection, observations)
└── Viewer (read-only access)
```

### Role-Based Access Control (RBAC)

```python
# Backend: app/core/permissions.py

from enum import Enum
from typing import List

class Role(str, Enum):
    ADMIN = "admin"
    BREEDER = "breeder"
    TECHNICIAN = "technician"
    VIEWER = "viewer"

class Permission(str, Enum):
    # Programs
    CREATE_PROGRAM = "create:program"
    READ_PROGRAM = "read:program"
    UPDATE_PROGRAM = "update:program"
    DELETE_PROGRAM = "delete:program"

    # Trials
    CREATE_TRIAL = "create:trial"
    READ_TRIAL = "read:trial"
    UPDATE_TRIAL = "update:trial"
    DELETE_TRIAL = "delete:trial"

    # Observations
    CREATE_OBSERVATION = "create:observation"
    READ_OBSERVATION = "read:observation"
    UPDATE_OBSERVATION = "update:observation"
    DELETE_OBSERVATION = "delete:observation"

    # Users (admin only)
    MANAGE_USERS = "manage:users"

# Role to permissions mapping
ROLE_PERMISSIONS: dict[Role, List[Permission]] = {
    Role.ADMIN: [
        # All permissions
        Permission.CREATE_PROGRAM, Permission.READ_PROGRAM,
        Permission.UPDATE_PROGRAM, Permission.DELETE_PROGRAM,
        Permission.CREATE_TRIAL, Permission.READ_TRIAL,
        Permission.UPDATE_TRIAL, Permission.DELETE_TRIAL,
        Permission.CREATE_OBSERVATION, Permission.READ_OBSERVATION,
        Permission.UPDATE_OBSERVATION, Permission.DELETE_OBSERVATION,
        Permission.MANAGE_USERS,
    ],
    Role.BREEDER: [
        # Can manage programs and trials
        Permission.CREATE_PROGRAM, Permission.READ_PROGRAM,
        Permission.UPDATE_PROGRAM,
        Permission.CREATE_TRIAL, Permission.READ_TRIAL,
        Permission.UPDATE_TRIAL,
        Permission.CREATE_OBSERVATION, Permission.READ_OBSERVATION,
        Permission.UPDATE_OBSERVATION,
    ],
    Role.TECHNICIAN: [
        # Can only collect field data
        Permission.READ_PROGRAM, Permission.READ_TRIAL,
        Permission.CREATE_OBSERVATION, Permission.READ_OBSERVATION,
        Permission.UPDATE_OBSERVATION,
    ],
    Role.VIEWER: [
        # Read-only access
        Permission.READ_PROGRAM, Permission.READ_TRIAL,
        Permission.READ_OBSERVATION,
    ],
}
```

## Future-Proofing for BrAPI Updates

### **CRITICAL REQUIREMENT: Design for BrAPI Version Flexibility**

When BrAPI v2.2 or v3.0 is released, Bijmantra must be able to support it without major rewrites.

### Architecture for Multi-Version Support

```python
# Backend structure for multi-version support

bijmantra/backend/
├── app/
│   ├── api/
│   │   ├── v2_0/          # BrAPI v2.0 (deprecated)
│   │   ├── v2_1/          # BrAPI v2.1 (current)
│   │   └── v2_2/          # BrAPI v2.2 (future)
│   ├── core/
│   │   ├── database.py    # Version-agnostic DB layer
│   │   └── models/        # Internal models (not BrAPI-specific)
│   └── adapters/          # BrAPI version adapters
│       ├── v2_0_adapter.py
│       ├── v2_1_adapter.py
│       └── v2_2_adapter.py
```

### Key Design Principles

#### 1. Separation of Concerns

```python
# Internal domain model (version-agnostic)
class Program:
    """Internal representation of a breeding program"""
    id: UUID
    name: str
    organization_id: UUID
    # ... internal fields

# BrAPI v2.1 adapter
class BrAPIv21ProgramAdapter:
    """Converts internal Program to BrAPI v2.1 format"""

    @staticmethod
    def to_brapi(program: Program) -> dict:
        """Convert internal model to BrAPI v2.1"""
        return {
            "programDbId": str(program.id),
            "programName": program.name,
            # ... BrAPI v2.1 fields
        }

    @staticmethod
    def from_brapi(data: dict) -> Program:
        """Convert BrAPI v2.1 to internal model"""
        return Program(
            id=UUID(data["programDbId"]),
            name=data["programName"],
            # ... map fields
        )

# BrAPI v2.2 adapter (future)
class BrAPIv22ProgramAdapter:
    """Converts internal Program to BrAPI v2.2 format"""
    # Same pattern, different mapping
```

#### 2. Version Negotiation

```python
# FastAPI version routing
from fastapi import APIRouter, Header

# BrAPI v2.1 router
router_v21 = APIRouter(prefix="/brapi/v2")

@router_v21.get("/programs")
async def get_programs_v21():
    programs = await get_programs_from_db()
    return [BrAPIv21ProgramAdapter.to_brapi(p) for p in programs]

# BrAPI v2.2 router (future)
router_v22 = APIRouter(prefix="/brapi/v2.2")

@router_v22.get("/programs")
async def get_programs_v22():
    programs = await get_programs_from_db()
    return [BrAPIv22ProgramAdapter.to_brapi(p) for p in programs]

# Include both routers
app.include_router(router_v21)
app.include_router(router_v22)  # When v2.2 is released
```

#### 3. Database Schema Versioning

```python
# Alembic migrations with version tracking

# Migration: 001_initial_schema.py
def upgrade():
    """Initial schema for BrAPI v2.1"""
    op.create_table(
        'programs',
        sa.Column('id', UUID, primary_key=True),
        sa.Column('name', sa.String(255)),
        # ... v2.1 fields
        sa.Column('brapi_version', sa.String(10), default='2.1'),
    )

# Migration: 015_add_v2_2_fields.py (future)
def upgrade():
    """Add fields for BrAPI v2.2 compatibility"""
    op.add_column('programs',
        sa.Column('new_v22_field', sa.String(255), nullable=True)
    )
```

#### 4. Feature Flags

```python
# app/core/config.py

class Settings(BaseSettings):
    # BrAPI version support

```

### Update Strategy When BrAPI v2.2 Releases

**Phase 1: Preparation (Before v2.2 release)**

1. Monitor BrAPI GitHub for v2.2 spec development
2. Review proposed changes
3. Assess impact on Bijmantra

**Phase 2: Implementation (After v2.2 release)**

1. Create new adapter layer for v2.2
2. Add new endpoints under `/brapi/v2.2/`
3. Keep v2.1 endpoints working (backward compatibility)
4. Update database schema if needed (migrations)
5. Test with BRAVA validator

**Phase 3: Migration**

1. Announce v2.2 support to users
2. Provide migration guide
3. Support both v2.1 and v2.2 for 1-2 years
4. Eventually deprecate v2.1 (with plenty of notice)
